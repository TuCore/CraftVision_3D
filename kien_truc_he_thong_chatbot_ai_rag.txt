TỔNG HỢP KIẾN TRÚC HỆ THỐNG CHATBOT AI (MULTIMODAL RAG)
=========================================================
Tech stack: Next.js (Frontend) + ASP.NET Core Web API (Backend) + PostgreSQL


1. KIẾN TRÚC TECH STACK
-------------------------

Frontend — Next.js
- Dùng App Router, gọi trực tiếp ASP.NET Core Web API (qua fetch) để nhận kết quả streaming.
- Không nên proxy qua Next.js API routes vì sẽ làm phức tạp việc forward streaming response
  không cần thiết.
- Luồng chat dạng "gõ chữ" (như ChatGPT): dùng ReadableStream (Server-Sent Events) từ
  ASP.NET Core, hoặc SignalR client nếu cần giao tiếp 2 chiều (vd: backend chủ động push
  trạng thái "đang xử lý video").
- Chức năng chính: giao diện chat, upload file (kéo thả ảnh/video), hiển thị kết quả streaming.

Backend — ASP.NET Core Web API (C#)
- Vai trò "nhạc trưởng" (Orchestrator): quản lý luồng dữ liệu, phân quyền, gọi AI API,
  giao tiếp với database qua Entity Framework Core.
- Dùng package Npgsql.EntityFrameworkCore.PostgreSQL làm EF Core provider cho PostgreSQL
  (thay vì provider SQL Server).
- Hỗ trợ trả response dạng stream ra Next.js qua IAsyncEnumerable hoặc minimal API trả về
  text/event-stream.

Database — PostgreSQL (gộp 2 vai trò nhờ pgvector)
- PostgreSQL đóng vai trò Relational Database: lưu Users, ChatSessions, ChatMessages,
  UploadedFiles, Quota (lượt chat).
- Cài thêm extension pgvector để PostgreSQL đóng luôn vai trò Vector Database: lưu
  embedding của ảnh và văn bản, dùng index HNSW hoặc IVFFlat để tìm kiếm ngữ nghĩa
  (semantic search) nhanh.
- Ưu điểm của việc gộp: chỉ cần vận hành 1 database engine thay vì 2 (không cần thêm
  Qdrant/Milvus/ChromaDB riêng) -> giảm chi phí hạ tầng và độ phức tạp khi deploy.
  Rất phù hợp với dự án vừa/nhỏ hoặc giai đoạn MVP.
- Khi nào vẫn nên tách Vector DB riêng: nếu tập dữ liệu vector lên tới hàng chục triệu
  bản ghi trở lên, cần độ trễ truy vấn cực thấp, hoặc cần tính năng vector-search nâng
  cao (hybrid search phức tạp, multi-tenant filtering ở quy mô lớn) -> khi đó Qdrant/Milvus
  vẫn tối ưu hơn pgvector. Với hầu hết ứng dụng (kể cả tính năng gợi ý quà tặng), pgvector
  là đủ dùng.

Object Storage (vẫn tách riêng, không đổi)
- AWS S3 / Azure Blob Storage / MinIO (chạy local).
- Lưu file ảnh, video gốc. PostgreSQL (kể cả có pgvector) KHÔNG phù hợp để lưu file
  ảnh/video gốc, nên Object Storage vẫn cần thiết như bình thường.


2. LUỒNG DỮ LIỆU (FRONTEND <-> BACKEND <-> DATABASE)
-------------------------------------------------------

A. Luồng tải file lên (Upload & Process)
   Next.js gửi:
   - HTTP POST multipart/form-data (file ảnh/video + User ID)
   Next.js nhận:
   - JSON trạng thái: đã xử lý xong hay đang trong hàng đợi (processing)

   ASP.NET Core gửi:
   - File gốc -> Object Storage
   - Metadata (tên file, URL, UserID, ngày tạo) -> PostgreSQL
   - Video được tách frame (FFmpeg) -> tạo Vector qua model Embedding -> lưu vào cột
     vector trong PostgreSQL (pgvector)
   ASP.NET Core nhận:
   - URL file từ Object Storage
   - ID bản ghi từ PostgreSQL
   - Xác nhận lưu vector thành công (pgvector)

B. Luồng hỏi đáp AI (Chat & RAG)
   Next.js gửi:
   - JSON: ChatSessionId, MessageText (câu hỏi)
   Next.js nhận:
   - Luồng text dạng stream (SSE hoặc SignalR) + URL ảnh trích xuất nếu có

   ASP.NET Core gửi:
   - Câu hỏi -> Vector -> truy vấn ngữ nghĩa trong PostgreSQL (pgvector)
   - Truy vấn PostgreSQL lấy lịch sử chat (bảng ChatMessages)
   - Đóng gói (câu hỏi + lịch sử + URL ảnh/video liên quan) -> gửi cho Vision AI
     (Gemini/GPT-4o...)
   ASP.NET Core nhận:
   - Vector tương đồng từ PostgreSQL (pgvector)
   - Văn bản trả lời dạng stream từ AI API


3. XỬ LÝ VIDEO & ẢNH CHO RAG
--------------------------------
- Ảnh: dùng model như CLIP để biến ảnh thành Vector toán học, lưu vào cột vector
  trong PostgreSQL (pgvector).
- Video: AI không đọc video thô. Backend cần:
  + Cắt video thành frame (ví dụ 1 giây/ảnh) bằng FFmpeg.
  + Tách audio -> chuyển thành transcript (speech-to-text).
  + Thực hiện RAG trên các frame và đoạn transcript này (lưu vector trong PostgreSQL).


4. LƯU Ý RIÊNG CHO TÍNH NĂNG "GỢI Ý QUÀ TẶNG TỰ LÀM"
---------------------------------------------------------

Luồng xử lý:
Input người dùng (text/ảnh + ngân sách + thời gian)
  -> Trích xuất đặc trưng + constraint (CLIP cho ảnh, parse ngân sách/thời gian)
  -> Truy vấn ngữ nghĩa trong PostgreSQL/pgvector (tìm ý tưởng quà, nguyên liệu,
     video tương tự)
  -> Đóng gói prompt (query + context tìm được + lịch sử chat)
  -> LLM sinh câu trả lời (Gemini/GPT/Claude...)
  -> Output có cấu trúc (danh sách quà, bảng nguyên liệu + giá, link mua, video hướng dẫn)

Các lưu ý kỹ thuật quan trọng:
- Nếu người dùng upload ảnh sản phẩm mẫu, cần dùng model vision (CLIP hoặc Vision API) để
  rút ra màu sắc, chất liệu, kỹ thuật thành mô tả text TRƯỚC khi nhúng vector.
  Không nên nhúng ảnh thô mà không có ngữ cảnh text đi kèm — việc tìm kiếm kiểu
  "đồ tái chế bằng len, dễ làm" sẽ kém chính xác nếu chỉ dựa vào vector ảnh thuần.

- Nên tách riêng một bảng/collection trong PostgreSQL chỉ chứa "công thức/tutorial quà
  tặng", gắn kèm metadata: độ khó, thời gian làm, dịp tặng, chi phí ước tính.
  Kết hợp metadata filter (WHERE) + vector search (pgvector) sẽ cho kết quả chính xác
  hơn là search thuần bằng vector.

- Output nên ép LLM trả về JSON theo schema cố định:
  + Tên quà tặng
  + Danh sách nguyên liệu kèm giá từng món
  + Tổng chi phí
  + Thời gian làm ước tính
  + Link mua nguyên liệu / từ khoá tìm kiếm
  + Từ khoá / link video hướng dẫn
  Sau đó ASP.NET Core Backend render JSON này thành bảng đẹp ở Next.js Frontend.
  Không nên để AI tự do format text vì sẽ khó parse chính xác thành bảng.


5. CÁC MÔ HÌNH AI (NGOÀI GEMINI 1.5 PRO) CÓ THỂ CÂN NHẮC
--------------------------------------------------------------

Lưu ý: Gemini 1.5 Pro đã khá cũ. Google hiện đã lên tới dòng Gemini 3.x
(Gemini 3.1 Pro là flagship, Gemini 3 Flash là bản giá rẻ hơn). Nên cân nhắc
dùng bản mới hơn nếu bắt đầu xây hệ thống mới.

a) Google AI Studio (Gemini) — vẫn là lựa chọn free tốt nhất cho multimodal
   - Free tier hỗ trợ đầy đủ input: ảnh, audio, video, tài liệu.
   - Không cần thẻ tín dụng, không hết hạn.
   - Nhược điểm: rate limit thấp (khoảng 5-15 request/phút tùy model)
     -> phù hợp để prototype, chưa phù hợp production tải cao.

b) Mistral (Le Chat / API)
   - Một trong hai hãng lớn (cùng Google) có free API tier.
   - Tuân thủ GDPR, lưu trữ dữ liệu tại EU.
   - Nhược điểm: context window chỉ 128K-256K token, không mạnh về
     sinh ảnh/video — vision chưa mạnh bằng Gemini.

c) DeepSeek
   - Mạnh về reasoning/code, giá rẻ, có bản chat miễn phí.
   - Vision (nhận diện ảnh) không phải thế mạnh chính
     -> chỉ nên dùng làm model phụ xử lý text/logic, không thay Gemini cho phần vision.

d) Các gateway trung gian (vd: AIHubMix và tương tự)
   - Cho phép gọi GPT-5.5, Gemini 3, GLM, Kimi... miễn phí qua 1 API key.
   - Là dịch vụ bên thứ ba subsidize chi phí, KHÔNG phải chính sách chính thức
     của OpenAI/Google -> độ ổn định và điều khoản có thể thay đổi.
   - Chỉ nên dùng để thử nghiệm nhanh, không dùng cho production nghiêm túc.

Gợi ý thực tế:
- Vì hệ thống cần nhận diện ảnh sản phẩm (màu sắc, chất liệu, kỹ thuật) và
  sinh output có cấu trúc phức tạp, Gemini free tier (Google AI Studio) vẫn là
  lựa chọn free mạnh nhất về vision hiện tại.
- Có thể dùng thêm Mistral hoặc DeepSeek làm model phụ cho tác vụ text thuần
  (viết mô tả, sinh gợi ý text) theo kiểu "split route" để giảm tải rate-limit
  của Gemini và không phụ thuộc hoàn toàn một nhà cung cấp.
